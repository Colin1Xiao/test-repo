# 📐 代码优化准则 (Code Refactoring Guidelines)

_小龙交易系统 - 生产代码优化规范_

---

## ⚠️ 核心原则

**优化不能破坏已有功能。**

任何优化如果导致功能失效，无论代码多漂亮，都是**失败**的优化。

---

## 🚫 历史教训 (2026-03-29)

### 问题
每次代码优化必定导致某些功能失效：
- 面板账户余额清零
- 数据写入异常
- 已完成功能被破坏

### 根因
1. 优化前未充分理解代码依赖关系
2. 缺少回归测试验证
3. 优化范围过大，一次性改动过多
4. 未保留回滚方案

## ✅ 优化流程 (必须遵守)

#### 1.1 功能清单 ✅
```markdown

```

#### 1.2 依赖分析 ✅
```markdown
- [ ] 调用方分析：谁在调用这个函数？
- [ ] 被调用方分析：这个函数依赖谁？
- [ ] 副作用分析：修改了哪些全局状态？
- [ ] 数据流分析：输入/输出数据结构
```

#### 1.3 测试准备 ✅
```markdown
- [ ] 准备验证脚本（验证核心功能）
- [ ] 记录优化前的关键数据（如账户余额）
- [ ] 准备回滚方案（git branch/备份）
```

#### 1.4 优化方案文档 ✅
```markdown
- [ ] 写清楚要优化什么
- [ ] 写清楚为什么要优化

- [ ] 写清楚回滚步骤
```

---

### 阶段 2: 小步优化

#### 2.1 单次优化范围限制
| 优化类型 | 最大范围 |
|---------|---------|
| 函数拆分 | 1 个函数/次 |
| 变量重命名 | 5 个变量/次 |
| 代码移动 | 1 个类/次 |
| 工具函数提取 | 3 个函数/次 |

#### 2.2 每次优化后必须验证
```bash
# 1. 语法检查
python3 -m py_compile panel_v40.py

# 2. 启动服务
python3 panel_v40.py &

# 3. 验证核心 API
curl http://localhost:8780/api/health
curl http://localhost:8780/api/snapshot

# 4. 验证关键数据
# - 账户余额 ≠ 0
# - 持仓数据正常
# - 历史数据完整
```

#### 2.3 验证清单 ✅
```markdown
- [ ] 语法检查通过
- [ ] 服务启动成功
- [ ] 核心 API 响应正常
- [ ] 关键数据未丢失（余额/持仓/历史）
- [ ] 日志无异常报错
```

---

### 阶段 3: 验证与回滚

#### 3.1 功能回归测试
```markdown
- [ ] 主面板数据显示正常
- [ ] 历史页数据加载正常
- [ ] 报表页统计正常
- [ ] 告警功能正常
- [ ] 数据更新正常（非静态数据）
```

#### 3.2 数据完整性验证
```markdown
- [ ] 账户余额 = 优化前值
- [ ] 持仓状态 = 优化前值
- [ ] 交易历史完整
- [ ] 日志文件正常写入
```

#### 3.3 回滚触发条件
**以下情况立即回滚：**
- ❌ 核心功能失效
- ❌ 数据丢失或损坏
- ❌ 无法定位的异常
- ❌ 验证时间 > 30 分钟

#### 3.4 回滚步骤
```bash
# 1. 停止服务
pkill -f "python3 panel_v40.py"

# 2. 回滚代码
git checkout panel_v40.py

# 3. 重启服务
python3 panel_v40.py &

# 4. 验证恢复
curl http://localhost:8780/api/health
```

## 🛠️ 安全优化技巧

### 1. 保持函数签名不变
```python
# ✅ 安全：签名不变，内部重构
def adapt_stats_payload(payload: Dict) -> Dict:
    # 内部调用新提取的辅助函数
    data = _extract_data_sources(payload)
    return {...}

# ❌ 危险：修改签名
def adapt_stats_payload(payload, new_param=None):  # 新增参数
    ...
```

### 2. 保持返回值结构不变
```python
# ✅ 安全：返回相同结构
return {
    "market": {...},
    "position": {...},
}

# ❌ 危险：修改返回结构
return {
    "market_data": {...},  # 改名
    "position": {...},
}
```

### 3. 保留向后兼容
```python
# ✅ 安全：保留旧函数名作为别名
from utils import safe_float as _safe_float

# ❌ 危险：直接删除旧函数
# 删除 _safe_float 定义，直接用 safe_float
```

### 4. 小步提交
```bash
# ✅ 安全：每步都可回滚
git commit -m "refactor: 提取数据源函数"
git commit -m "refactor: 提取计算函数"
git commit -m "refactor: 提取视图函数"

# ❌ 危险：一次性提交
git commit -m "refactor: 优化 adapt_stats_payload"
```

## 🎯 优化优先级

### 高优先级（值得优化）
1. **重复代码** - 多处复制粘贴
2. **超长函数** - >100 行且逻辑复杂
3. **魔法数字** - 未命名的常量
4. **模糊命名** - 变量/函数名不清晰

### 低优先级（谨慎优化）
1. **代码风格** - 个人偏好差异
2. **性能微调** - <10% 提升
3. **重构架构** - 大规模改动

### 禁止优化
1. **正在运行的生产代码** - 无充分测试
2. **核心交易逻辑** - 无完整回归测试
3. **紧急修复期间** - 优先解决问题

## 🔒 红线（绝对禁止）

1. **无测试不优化** - 没有验证脚本不动手
2. **无备份不优化** - 没有回滚方案不动手
3. **无文档不优化** - 没有优化方案不动手
4. **生产时段不优化** - 交易时间不动手
5. **疲劳状态不优化** - 深夜/困倦时不动手
