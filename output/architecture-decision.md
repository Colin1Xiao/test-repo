# 电商平台重构架构决策报告

## 一、最终推荐架构

**演进式微服务架构（Evolutionary Microservices）**

采用"模块化单体 → 微服务"的两阶段演进路径，而非一步到位实施微服务。

---

## 二、关键理由

### 2.1 为什么不是一步到位微服务？

| 约束条件 | 微服务直接落地的挑战 |
|---------|---------------------|
| 10人团队 | 微服务需要运维、DevOps、监控等专项人力，小团队难以覆盖 |
| 快速上线 | 微服务基础设施搭建需3-6个月，无法满足时间压力 |
| 成本可控 | 微服务需要服务网格、分布式追踪、多环境部署，成本倍增 |
| 团队技能 | 分布式系统调试、数据一致性处理需要较高技术门槛 |

### 2.2 为什么不是纯单体架构？

| 长期要求 | 纯单体的隐患 |
|---------|-------------|
| 100万用户规模 | 单体扩展存在天花板，数据库连接池、代码复杂度将成为瓶颈 |
| 易于维护 | 随着业务增长，单体代码耦合度会指数级上升 |
| 高并发要求 | 无法针对热点模块（如支付）单独扩容 |

### 2.3 演进式架构的核心优势

```
┌─────────────────────────────────────────────────────────┐
│                    演进式微服务架构                      │
├─────────────────────────────────────────────────────────┤
│  第一阶段（3-6个月）        →      第二阶段（6-12个月）   │
│  ┌─────────────────┐              ┌─────────────────┐   │
│  │   模块化单体     │              │   关键服务拆分   │   │
│  │  • 用户模块      │    演进      │  • 支付服务      │   │
│  │  • 订单模块      │   ─────→     │  • 订单服务      │   │
│  │  • 支付模块      │              │  • 网关层        │   │
│  │  • 库存模块      │              │  • 配置中心      │   │
│  └─────────────────┘              └─────────────────┘   │
│         ↓                              ↓                │
│    快速上线能力                   独立扩展能力            │
│    低成本运维                     高并发支撑              │
│    团队熟悉度                     故障隔离                │
└─────────────────────────────────────────────────────────┘
```

**核心逻辑：**
1. **先跑通，再优化** — 用单体速度验证业务，用模块化设计预留拆分空间
2. **先核心，再边缘** — 支付/订单先拆分，用户/库存后跟进
3. **先工具，再平台** — 从简单的服务注册发现开始，逐步完善基础设施

## 三、实施建议

### 3.1 第一阶段：模块化单体（Month 1-6）

#### 技术栈推荐

| 层级 | 技术选型 | 理由 |
|-----|---------|-----|
| 后端框架 | Spring Boot / Go + Gin | 团队熟悉、生态成熟、性能优秀 |
| 数据库 | MySQL（主）+ Redis（缓存） | 成本可控、运维简单 |
| 消息队列 | RabbitMQ / RocketMQ | 异步解耦、削峰填谷 |
| 接口规范 | RESTful + OpenAPI | 标准化、便于后续拆分 |

#### 模块划分原则

```
com.ecommerce
├── user-module/          # 用户模块（独立包）
│   ├── api/              # 对外接口
│   ├── service/          # 业务逻辑
│   └── repository/       # 数据访问
├── order-module/         # 订单模块（独立包）
├── payment-module/       # 支付模块（独立包）
├── inventory-module/     # 库存模块（独立包）
└── shared/               # 公共组件
    ├── common/           # 工具类
    └── event/            # 领域事件
```

**关键约束：**
- 模块间禁止直接调用DAO/Repository
- 必须通过Service接口或领域事件通信
- 每个模块有独立的单元测试套件

#### 数据架构

```
┌─────────────────────────────────────┐
│           单体数据库                 │
│  ┌─────────┐ ┌─────────┐            │
│  │ user_db │ │order_db │            │
│  │ schema  │ │ schema  │            │
│  └─────────┘ └─────────┘            │
│  ┌─────────┐ ┌─────────┐            │
│  │pay_db   │ │stock_db │            │
│  │ schema  │ │ schema  │            │
│  └─────────┘ └─────────┘            │
└─────────────────────────────────────┘
              ↓
    逻辑分离，物理共用（为后续拆分做准备）
```

### 3.2 第二阶段：关键服务拆分（Month 6-12）

#### 拆分优先级

```
优先级排序：
1. 支付服务（最高）— 安全要求高、变更频率低、需要独立部署
2. 订单服务（高）— 业务核心、并发量大、需要独立扩容
3. 库存服务（中）— 数据一致性要求高
4. 用户服务（低）— 相对稳定、可最后拆分
```

#### 微服务基础设施（逐步引入）

| 组件 | 选型建议 | 引入时机 |
|-----|---------|---------|
| API网关 | Spring Cloud Gateway / Nginx | 第二阶段初期 |
| 服务注册 | Nacos / Consul | 第二阶段初期 |
| 配置中心 | Nacos / Apollo | 第二阶段初期 |
| 监控告警 | Prometheus + Grafana | 第二阶段中期 |
| 链路追踪 | Jaeger / Zipkin | 第二阶段中期 |
| 日志收集 | ELK Stack / Loki | 第二阶段后期 |

#### 数据一致性方案

```
订单创建流程（分布式事务）：

1. 订单服务创建订单（本地事务）
        ↓
2. 发送订单创建事件到消息队列
        ↓
3. 库存服务消费事件，扣减库存
        ↓
4. 支付服务消费事件，创建支付单
        ↓
5. 各服务异步确认，失败则补偿

核心原则：最终一致性 + 补偿机制
```

### 3.3 上线策略

#### 分阶段发布计划

```
Month 1-2:  用户模块重构 + 灰度发布（5%流量）
Month 3-4:  订单模块重构 + 灰度发布（20%流量）
Month 5-6:  支付/库存模块 + 全量发布
Month 7-9:  支付服务拆分 + 独立部署
Month 10-12: 订单服务拆分 + 完善监控
```

#### 风险控制措施

1. **数据库双写**：新旧系统并行运行，数据同步校验
2. **灰度开关**：功能级别开关，随时回滚
3. **监控埋点**：核心链路100%覆盖，异常自动告警
4. **应急预案**：数据不一致时的修复脚本、回滚方案

### 3.4 团队组织建议

```
10人团队分工：

┌─────────────────────────────────────────┐
│  架构组（2人）                           │
│  • 整体技术方案、代码审查、性能优化        │
├─────────────────────────────────────────┤
│  后端开发组（5人）                        │
│  • 用户/订单模块（2人）                   │
│  • 支付模块（2人）— 独立部署后专职         │
│  • 库存模块（1人）                        │
├─────────────────────────────────────────┤
│  质量保障组（2人）                        │
│  • 自动化测试、CI/CD、灰度发布            │
├─────────────────────────────────────────┤
│  运维支持（1人）                          │
│  • 监控、告警、基础设施维护                │
└─────────────────────────────────────────┘
```

### 3.5 成本估算

| 阶段 | 主要成本项 | 预估范围 |
|-----|-----------|---------|
| 第一阶段 | 云服务器（4核8G×3）+ MySQL + Redis | 3-5万/年 |
| 第二阶段 | 增加微服务组件 + 监控 + 扩容 | 8-12万/年 |
| 人力成本 | 10人团队 × 12个月 | 按实际计算 |

---

## 四、关键成功因素

1. **模块边界清晰** — 前期多花时间在领域建模上，避免后期拆分困难
2. **自动化测试覆盖** — 重构的底气来自测试，目标80%+覆盖率
3. **监控先行** — 上线前必须可观测，否则问题定位将是一场灾难
4. **渐进式演进** — 不要追求完美架构，先解决当前痛点
5. **文档沉淀** — 架构决策记录（ADR）、接口文档、运维手册

---

## 五、总结

**推荐架构：演进式微服务（模块化单体 → 微服务）**

这不是妥协，而是基于约束条件的最优解：
- 它让10人团队能在6个月内交付可用系统
- 它为100万用户规模预留了扩展空间
- 它将微服务的复杂度分散到12个月逐步消化
- 它保持了成本可控和系统可维护性

**核心口诀：先模块化，再服务化；先跑通，再优化；先核心，再边缘。**
