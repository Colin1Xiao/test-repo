# OpenClaw 第二阶段优化路线图

**阶段：** 正式运行后持续优化  
**目标：** 从"能用"到"更稳、更省、更容易运维"  
**时间：** 2026-03-13 起持续迭代

---

## 一、必做（稳定性与可运维性）

### 1. 自动告警机制 ⭐⭐⭐

**目标：** 从"可观测"到"可运维"

**告警条件：**
- [ ] 成功率低于 95%
- [ ] 某模型 timeout 率 > 10%
- [ ] empty_response 连续出现 3 次
- [ ] fallback 触发数异常升高（> 20%）
- [ ] MAIN 汇总耗时 > 60s
- [ ] 某条混合链路连续失败 2 次

**告警渠道：**
- [ ] Telegram 通知
- [ ] 日志标记 ERROR
- [ ] 每日报告高亮

**优先级：** P0

---

### 2. 自动重试策略细化 ⭐⭐⭐

**目标：** 不同错误不同处理，不一刀切

**错误分类处理：**

| 错误类型 | 可重试 | 重试次数 | 退避策略 | fallback |
|----------|--------|----------|----------|----------|
| empty_response | ✅ | 1 | 立即 | 允许 |
| provider_error | ✅ | 2 | 2s 退避 | 允许 |
| provider_slow_response | ✅ | 1 | 5s 退避 | 允许 |
| timeout | ❌ | 0 | - | 必须 |
| subagent_invoke_failed | ✅ | 1 | 立即 | 允许 |

**实现：**
- [ ] 错误分类器
- [ ] 重试计数器
- [ ] 退避延时器

**优先级：** P0

---

### 3. GROK-CODE 长期观测 ⭐⭐⭐

**目标：** 判断是偶发抖动还是长期长尾

**观测内容：**
- [ ] 每日稳定性报告
- [ ] 异常样本自动归档
- [ ] P95 趋势图
- [ ] 空输出率趋势

**判断阈值：**
- P95 > 40s 连续 3 天 → 考虑调 timeout
- 空输出率 > 5% 连续 3 天 → 检查 provider

**优先级：** P1

---

### 4. 结果质量抽样 ⭐⭐⭐

**目标：** 验证"成功返回"是否"值得用"

**抽样策略：**
- [ ] 每类任务每日抽样 5-10 条
- [ ] 人工/自动评估质量

**评估维度：**

| 模型 | 评估点 |
|------|--------|
| FAST | 是否过度简略 |
| LONG | 结构是否稳定 |
| GROK-CODE | 是否真能定位根因 |
| REASON | 是否有推理漂移 |
| CN | 是否过润色或跑偏 |
| MAIN | 是否压缩过头 |

**优先级：** P1

---

## 二、推荐做（可扩展性与成本）

### 5. 回退策略正式化 ⭐⭐

**目标：** "降级但不中断"

**回退规则：**

| 原链路 | 失败时回退到 |
|--------|--------------|
| GROK-CODE → CODE → MAIN | CODE + MAIN 或 MAIN 直接诊断 |
| LONG → REASON → MAIN | MAIN 直接基于 LONG 摘要输出 |
| REASON → CN → MAIN | 保留未润色版本直接返回 |

**实现：**
- [ ] 回退决策器
- [ ] 降级结果生成器

**优先级：** P2

---

### 6. 任务类型白名单/黑名单 ⭐⭐

**目标：** 减少不必要复杂度和成本

**规则：**

| 任务特征 | 处理方式 |
|----------|----------|
| 很短的简单问题 (< 50字) | 强制单模型，不走混合链路 |
| 极长任务 (> 5000字输入) | 先强制摘要再编排 |
| 高不确定任务 | 直接交 MAIN 兜底 |
| 纯代码生成 | 强制 CODE/CODE-PLUS |
| 纯中文润色 | 强制 CN |

**实现：**
- [ ] 任务特征分类器
- [ ] 路由前置过滤器

**优先级：** P2

---

### 7. 成本统计 ⭐⭐

**目标：** 识别高成本低收益任务

**统计项：**
- [ ] 每请求成本估算
- [ ] 每模型成本占比
- [ ] 每条混合链路平均成本
- [ ] 高耗时高成本低收益任务识别

**实现：**
- [ ] 成本计算器（基于耗时和模型）
- [ ] 成本报告生成器

**优先级：** P3

---

### 8. 运行手册 ⭐⭐

**目标：** 从"你能管"到"别人也能接手"

**文档内容：**
- [ ] 出现空输出怎么办
- [ ] 出现 provider 波动怎么办
- [ ] 何时调 timeout
- [ ] 何时回滚
- [ ] 何时扩大范围
- [ ] 何时发布新基线

**优先级：** P2

---

## 三、暂缓做（未来扩展）

| 项目 | 原因 |
|------|------|
| 立刻改更多模型 | 先稳定运行积累数据 |
| 立刻扩大开放范围 | 保持 allowlist 观察 |
| 立刻再调 timeout | 基于数据而非猜测 |
| 更复杂的多层编排 | 当前 3 步已够用 |
| 全自动自治 | 先有人工兜底 |

### 第 4 周（推荐）
6. 任务类型白名单/黑名单
7. 运行手册

### 第 2 个月（成本优化）
8. 成本统计

## 五、当前阶段判断

| 维度 | 当前状态 | 下一步目标 |
|------|----------|------------|
| 功能完整性 | ✅ 100% | 保持，不急于扩展 |
| 稳定性 | ✅ 良好 | 告警+重试细化 → 优秀 |
| 可运维性 | ⚠️ 中等 | 告警+手册 → 良好 |
| 可观测性 | ✅ 优秀 | 质量抽样 → 更精准 |
| 成本意识 | ❌ 暂无 | 成本统计 → 有 |
| 团队可接手 | ⚠️ 中等 | 运行手册 → 良好 |

---

## 六、一句话结论

> 还可以继续调，也值得再增添一些运维和降级能力，但已经不是"功能缺口"，而是"正式运行后的稳定性、可运维性和可扩展性增强"。

**最优先三件事：**
1. 自动告警（从可观测到可运维）
2. 重试细化（不同错误不同处理）
3. 质量抽样（验证成功是否值得用）
