## 回退策略总览

**目标：** 把"有 fallback"变成"什么情况下怎么降级"有清晰规则

**核心原则：**
- 降级但不中断
- 保留已完成的部分
- 给用户明确的说明

## 混合任务链路回退规则

### 链路 1：GROK-CODE → CODE → MAIN

| 失败步骤 | 回退策略 | 用户收到 |
|----------|----------|----------|
| GROK-CODE 失败 | 降级到 CODE 直接诊断 | CODE 生成的修复方案 + 说明 |
| CODE 失败 | 返回 GROK-CODE 诊断结果 | 仅诊断分析，无代码修复 |
| MAIN 失败 | 返回前两步摘要 | 诊断 + 修复摘要 |

**回退路径：**
```
正常: GROK-CODE → CODE → MAIN
      ↓
GROK-CODE 失败: CODE (直接诊断)
      ↓
CODE 失败: GROK-CODE 结果
      ↓
MAIN 失败: 前两步摘要
```

---

### 链路 2：LONG → REASON → MAIN

| 失败步骤 | 回退策略 | 用户收到 |
|----------|----------|----------|
| LONG 失败 | 直接交 REASON 处理原始输入 | REASON 基于原始输入的推理 |
| REASON 失败 | 返回 LONG 摘要 | 文档摘要 + 说明 |
| MAIN 失败 | 返回前两步摘要 | 摘要 + 分析摘要 |

**回退路径：**
```
正常: LONG → REASON → MAIN
      ↓
LONG 失败: REASON (原始输入)
      ↓
REASON 失败: LONG 摘要
      ↓
MAIN 失败: 前两步摘要
```

---

### 链路 3：REASON → CN → MAIN

| 失败步骤 | 回退策略 | 用户收到 |
|----------|----------|----------|
| REASON 失败 | 直接返回未润色版本 | 原始分析结果 |
| CN 失败 | 返回 REASON 结果 | 未润色的分析 |
| MAIN 失败 | 返回前两步摘要 | 分析 + 润色摘要 |

**回退路径：**
```
正常: REASON → CN → MAIN
      ↓
REASON 失败: 返回原始输入说明
      ↓
CN 失败: REASON 结果
      ↓
MAIN 失败: 前两步摘要
```

## 异常类型回退规则

### Provider Error

| 异常类型 | 处理策略 | 回退目标 |
|----------|----------|----------|
| provider_slow_response | 重试 1 次，仍失败则 fallback | 同链路 fallback |
| provider_error (5xx) | 重试 2 次，仍失败则 fallback | 同链路 fallback |
| provider_error (4xx) | 不重试，直接 fallback | 同链路 fallback |
| timeout | 不重试，直接 fallback | 同链路 fallback |

### Empty Response

| 场景 | 处理策略 | 回退目标 |
|------|----------|----------|
| empty_response | 重试 1 次，仍失败则 fallback | 同链路 fallback |
| assistant_stub_only | 标记为 empty，重试 1 次 | 同链路 fallback |

## 回退决策矩阵

```
检测到失败
    ↓
判断失败类型
    ↓
┌─────────────────┬─────────────────┐
│ 可重试错误       │ 不可重试错误     │
│ (empty/slow)    │ (timeout/4xx)   │
└────────┬────────┴────────┬────────┘
         ↓                 ↓
    执行重试          直接 fallback
    (1-2次)           (不重试)
         ↓                 ↓
    重试成功?         执行 fallback
         ↓                 ↓
    是 → 继续        返回降级结果
    否 → fallback
```

## 用户侧降级返回模板

### 模板 1：部分完成（有成功步骤）

```
【部分完成结果】

✅ 已完成：
{已完成步骤摘要}

⚠️ 未完成：
{失败步骤说明}

💡 建议：
1. 基于已完成的内容，您可以...
2. 针对失败部分，建议...
```

### 模板 2：完全降级（全部失败）

```
【服务降级通知】

当前请求遇到技术问题，已自动降级处理。

📋 问题说明：
{失败原因}

🔧 当前状态：
系统已切换至备用方案，但结果可能不完整。

💡 建议：
1. 稍后重试
2. 简化问题后重试
3. 联系管理员
```

### 模板 3：单步失败（混合链路中）

```
【步骤 {step_index} 失败】

前序步骤已完成，但 {model} 步骤遇到问题。

✅ 已完成：
{前序结果摘要}

⚠️ 失败步骤：{step_name}
原因：{error_type}

💡 建议：
可基于已完成的部分继续，或针对失败步骤单独重试。
```

## 回退策略实现要点

### 1. 状态传递

- 每个步骤必须返回：成功/失败状态 + 输出/错误
- 失败时携带：错误类型、重试次数、建议回退路径

### 2. 结果保留

- 成功步骤的结果必须保留，不因后续失败而丢失
- 结果存储在上下文（context）中，供后续 fallback 使用
