## 🔴 高风险 (Critical) - 需立即处理

### 1. Gateway 单点故障

**风险等级:** 🔴 Critical  
**影响范围:** 100% 系统功能  
**发生概率:** 中 (依赖进程稳定性)

**风险描述:**
```
OpenClaw Gateway 是整个系统的唯一入口：
- Gateway 停止 → Telegram 断连
- Gateway 停止 → API 不可用
- Gateway 停止 → 技能系统瘫痪
- Gateway 停止 → Auto-Heal 无法触发
```

**当前状态:**
```json
{
  "gateway": {
    "status": "running",
    "port": 18789,
    "severity": 0
  }
}
```

**现有保护:**
- ✅ 健康检查脚本监控 (`openclaw-health-check.sh`)
- ✅ Auto-Heal 可触发 `gateway_restart` (需审批)
- ✅ 状态文件实时更新 (`openclaw-health-check.json`)

**缺失保护:**
- ❌ 无热备机制 (双活/主备)
- ❌ 无自动重启 (需 Auto-Heal 介入)
- ❌ 无看门狗进程 (Gateway 崩溃后无法自启)
- ❌ 无日志异常模式检测 (崩溃前无预警)

**改进建议:**
```
P0 [立即]: 实现 Gateway 看门狗进程
  - 使用 launchd (macOS) 守护进程
  - 崩溃后自动重启 (最大重试 3 次)
  - 重启失败后发送告警

P0 [本周]: 设计 Gateway 双活方案
  - 方案 A: 双 Gateway 实例 + 负载均衡
  - 方案 B: 热备实例 (主实例失败后自动切换)
  - 方案 C: 轻量级备用入口 (仅关键功能)

P1 [本月]: Gateway 日志异常检测
  - 检测模式：内存泄漏、连接数激增、响应延迟
  - 提前预警：崩溃前 5 分钟发送告警
```

**验收标准:**
- [ ] Gateway 崩溃后 30 秒内自动重启
- [ ] 连续 3 次重启失败 → 发送 Critical 告警
- [ ] 双活方案：单实例故障不影响 Telegram 消息收发

### 2. 交易系统：状态一致性风险 (已部分缓解)

**风险等级:** 🟡 Warning → 🟢 Mitigated (V5.4)  
**影响范围:** 交易决策、持仓状态、盈亏统计  
**发生概率:** 低 (V5.4 已修复)

**风险描述 (V5.3):**
```
- 多个 Python 进程同时访问 state_store.json
- 无文件锁机制 → 竞态条件导致状态不一致
- 备份文件众多暗示存在并发写风险:
  - state_store.py.bak_before_phase1c_fix_v4
  - live_executor.py.bak
  - execution_engine.py.bak_before_phase1c_fix_v2
```

**V5.4 修复措施:**
```python
# ✅ 文件锁保护 (state_store_v54.py)
async def record_event(self, event_type: str, data: dict):
    with filelock.FileLock(self._lock_path):
        # 原子写入
        with open(self._path, 'w') as f:
            json.dump(data, f)

# ✅ 执行锁 (safe_execution_v54.py)
async def execute_entry(self, ctx: ExecutionContext):
    async with self._execution_lock:  # asyncio.Lock
        # 同一时间只允许 1 个执行线程
        ...
```

**剩余风险:**
- ⚠️ 文件锁依赖 `fcntl` (macOS/Linux 兼容，Windows 不兼容)
- ⚠️ 锁超时未处理 (死锁风险)
- ⚠️ 未迁移到 SQLite (已有 `storage_sqlite.py` 但未全面使用)

**改进建议:**
```
P1 [本周]: 锁超时保护
  - 文件锁超时：5 秒
  - 执行锁超时：10 秒 (已实现)
  - 超时后发送告警 + 回滚

P2 [本月]: 迁移到 SQLite
  - 优势：事务支持、并发安全、查询性能
  - 已有基础：storage_sqlite.py
  - 迁移路径：双写 → 验证 → 切换

P2 [本月]: 状态一致性校验
  - 启动时校验：本地状态 vs 交易所状态
  - 定期巡检：每 5 分钟校验一次
  - 不一致时：告警 + 自动修复建议
```

**验收标准:**
- [ ] 并发双信号 → 只成功 1 笔 (另一笔被 Position Gate 挡住)
- [ ] 状态文件损坏率：0% (1000 次并发测试)
- [ ] SQLite 迁移后：性能提升 >20%，无数据丢失

### 3. 技能系统：无沙箱隔离

**风险等级:** 🔴 Critical  
**影响范围:** 文件系统、网络、系统命令  
**发生概率:** 低 (但有技能审查机制)

**风险描述:**
```
47+ 个技能直接运行在主机环境：
- 技能代码有完整文件系统访问权限
- 技能可执行任意 shell 命令 (exec 工具)
- 技能可访问网络 (web_search, web_fetch, browser)
- 恶意/错误技能可能破坏系统

现有保护:
- skill-vetter 技能审查机制 (安装前检查)
- behavior-guard.json 行为边界控制
- 但：运行时完全无限制
```

**攻击场景:**
```
1. 恶意技能安装 → 读取 ~/.ssh/id_rsa
2. 恶意技能安装 → 删除 workspace 文件
3. 恶意技能安装 → 发送敏感数据到外部
4. 技能 bug → 无限循环消耗 CPU/内存
5. 技能 bug → 写入大文件耗尽磁盘
```

**改进建议:**
```
P0 [立即]: 技能权限分级
  - 只读技能：read, web_search, web_fetch
  - 写入技能：write, edit (需 workspace 内)
  - 执行技能：exec (需审批)
  - 网络技能：browser, message (需配置)

P1 [本周]: 技能沙箱化 (NSJail/Docker)
  - 文件系统隔离：仅允许访问技能目录 + workspace
  - 网络隔离：仅允许访问配置的域名
  - 资源限制：CPU 50%, 内存 512MB, 时间 60s

P1 [本周]: 技能执行审计日志
  - 记录：技能名、工具调用、参数、结果
  - 告警：敏感操作 (文件删除、网络请求)
  - 追溯：安全事件可回溯

P2 [本月]: 技能资源限制
  - CPU 限制：每个技能 ≤50%
  - 内存限制：每个技能 ≤512MB
  - 时间限制：每个技能 ≤60s
  - 文件限制：写入 ≤10MB/次
```

**验收标准:**
- [ ] 技能无法访问 `~/.ssh/` 目录
- [ ] 技能无法删除 workspace 外文件
- [ ] 技能执行超时 60s 自动终止
- [ ] 审计日志记录所有 exec 调用
