## 1. 系统简介

OpenClaw 是一个多模型 AI 编排系统，支持 8 个模型智能路由，可根据任务类型自动选择最适合的模型或模型组合。

### 核心能力

- **8 模型支持**：MAIN, FAST, LONG, CODE, CODE-PLUS, REASON, CN, GROK-CODE
- **智能路由**：单模型直达或混合任务编排
- **自动告警**：8 条告警规则，主动发现问题
- **故障恢复**：自动降级，保证服务可用性

---

## 2. 系统架构

```
用户请求
    ↓
路由决策器
    ↓
┌─────────────┬─────────────┐
│  单模型      │  混合任务    │
│  直达        │  编排       │
└─────────────┴─────────────┘
    ↓
模型执行
    ↓
结果返回
```

### 模型职责

| 模型 | 职责 | 典型任务 |
|------|------|----------|
| FAST | 快速响应 | 简单问答、事实查询 |
| LONG | 长文本处理 | 文档总结、多文档整合 |
| CODE | 代码生成 | 函数、脚本、接口 |
| CODE-PLUS | 大型代码 | 重构、多文件调整 |
| GROK-CODE | 代码诊断 | Debug、根因分析 |
| REASON | 复杂推理 | 方案比较、决策分析 |
| CN | 中文优化 | 润色、改写、正式化 |
| MAIN | 总控兜底 | 汇总、兜底、图片 |

## 4. 故障处理

### 4.1 出现空输出怎么办

**现象：** 模型返回空或只有 "Assistant:"

**处理步骤：**
1. 检查是否触发告警（ALERT-003）
2. 查看对应模型 provider 状态
3. 系统自动重试 1 次
4. 如仍失败，进入 fallback
5. 记录异常样本

**预期：** 用户收到降级结果，服务不中断

### 4.2 出现 Provider 波动怎么办

**现象：** 某模型频繁 timeout 或 error

**处理步骤：**
1. 检查告警（ALERT-002, ALERT-008）
2. 查看 provider 服务状态页
3. 系统自动 fallback 到替代模型
4. 如持续 30 分钟，考虑临时禁用该模型

**临时禁用命令：**
```python
# 在配置中临时注释该模型
# 重启 gateway
openclaw gateway restart
```

### 4.3 何时调 Timeout

**GROK-CODE：**
- P95 > 40s 连续 3 天 → 调至 60s
- 空输出率 > 5% 连续 3 天 → 检查 provider

**其他模型：**
- 一般不调，优先优化输入或换模型

### 4.4 何时回滚

**触发条件：**
- 高频空输出重复出现
- 关键混合链路连续失败
- MAIN 汇总普遍超时
- 某模型 provider 持续异常

**回滚步骤：**
1. 切换到上一稳定基线
2. 暂时关闭高风险混合链路
3. 保留核心单模型能力
4. 通知用户服务降级

## 7. 常见问题 FAQ

**Q: 系统响应慢怎么办？**
A: 检查是否触发 MAIN 汇总超时告警，查看输入长度是否过大。

**Q: 某模型一直失败怎么办？**
A: 检查该模型 provider 状态，必要时临时禁用或切换 fallback。

**Q: 如何查看某条请求的处理过程？**
A: 通过 request_id 查询日志：grep "request_id" logs/requests.log

**Q: 质量抽样结果不好怎么办？**
A: 记录具体问题，调整路由策略或模型参数。

**Q: 告警太多怎么办？**
A: 检查阈值设置是否合理，调整去重时间，区分 WARNING/CRITICAL。
